\hypertarget{md_README_autotoc_md2}{}\doxysection{\texorpdfstring{Descripción}{Descripción}}\label{md_README_autotoc_md2}
El jugador (o jugadores) exploran un castillo compuesto por salas conectadas mediante {\bfseries{enlaces (Links)}} bidireccionales que pueden estar abiertos o cerrados. Es posible recoger objetos, inspeccionarlos, hablar con personajes amistosos y combatir contra enemigos en un sistema de combate por turnos con resultado aleatorio.

El motor gráfico renderiza en ASCII la sala actual junto con las cuatro salas adyacentes (N/\+\+S/\+\+E/\+W), el inventario del jugador activo, las barras de salud y los mensajes de los personajes.

\hypertarget{md_README_autotoc_md4}{}\doxysection{\texorpdfstring{Características destacadas (I3)}{Características destacadas (I3)}}\label{md_README_autotoc_md4}

\begin{DoxyItemize}
\item {\bfseries{Multijugador por turnos}} — varios jugadores rotan en orden; cada turno pertenece a un jugador distinto.
\item {\bfseries{Links bidireccionales}} — cada enlace almacena el estado abierto/\+cerrado de forma independiente por sentido de travesía ({\ttfamily orig→dest} y {\ttfamily dest→orig}), lo que permite pasajes de un solo sentido.
\item {\bfseries{Descubrimiento de salas (F12)}} — las salas desconocidas se muestran como cajas vacías hasta que el jugador las visita; una vez descubiertas se revelan su {\ttfamily gdesc}, objetos y personajes.
\item {\bfseries{Sistema de combate PvP}} — {\ttfamily attack} funciona tanto contra NPCs enemigos como contra otros jugadores en la misma sala; el autoataque queda vetado.
\item {\bfseries{Motor gráfico enriquecido}} — renderiza las 4 salas vecinas, barras de salud ASCII ({\ttfamily \+[\#\#\#-\/-\/]\+ 3 !}), etiquetas {\ttfamily \+[ALLY]\+}/\+{\ttfamily \+[ENEMY]\+} en personajes y borde de color por jugador activo.
\item {\bfseries{Log de partida (F14)}} — bandera {\ttfamily -\/l \texorpdfstring{$<$}{<}fichero\texorpdfstring{$>$}{>}} que registra cada comando con su resultado {\ttfamily OK}/\+{\ttfamily ERROR}.
\item {\bfseries{Inventario multi-\/objeto}} — el jugador puede llevar varios objetos simultáneamente; {\ttfamily inspect} comprueba tanto el inventario como la sala actual.
\item {\bfseries{Memoria limpia}} — 0 leaks/\+errores Valgrind, 0 advertencias con {\ttfamily -\/Wall -\/Wextra -\/Wpedantic}.
\end{DoxyItemize}

El movimiento usa {\ttfamily Links}:\+ se busca el link que une la sala actual con la sala destino en la dirección indicada. Si el link existe y {\ttfamily open\+\_\+orig\+\_\+to\+\_\+dest == TRUE}, el movimiento se autoriza.\hypertarget{md_README_autotoc_md22}{}\doxysubsection{\texorpdfstring{Combate ({\ttfamily attack})}{Combate ({\ttfamily attack})}}\label{md_README_autotoc_md22}
Se lanza un dado 0–9:\+
\begin{DoxyItemize}
\item {\bfseries{0–4}} → el enemigo gana el intercambio:\+ el jugador activo pierde 1 HP.
\item {\bfseries{5–9}} → el jugador gana:\+ el objetivo pierde 1 HP.
\end{DoxyItemize}

Si la salud del jugador activo llega a 0, la partida termina. El PvP sigue las mismas reglas; el autoataque retorna {\ttfamily ERROR}.\hypertarget{md_README_autotoc_md23}{}\doxysubsection{\texorpdfstring{Chat}{Chat}}\label{md_README_autotoc_md23}

\hypertarget{md_README_autotoc_md37}{}\doxysection{\texorpdfstring{Notas de diseño}{Notas de diseño}}\label{md_README_autotoc_md37}

\begin{DoxyItemize}
\item El módulo {\ttfamily entity} proporciona una base común (id + nombre) para {\ttfamily Player}, {\ttfamily Object} y {\ttfamily Character}, reduciendo la duplicación de código.
\item {\ttfamily Set} e {\ttfamily Inventory} desacoplan la gestión de colecciones del resto de módulos; {\ttfamily Inventory} añade la capa de límite de capacidad sobre {\ttfamily Set}.
\item Los {\bfseries{Links}} separan la topología del mapa de los propios espacios, permitiendo puertas con estado abierto/\+cerrado {\bfseries{independiente por dirección}} (extensión bidireccional consciente sobre el enunciado).
\item El motor gráfico ({\ttfamily graphic\+\_\+engine}) está deliberadamente desacoplado del estado del juego ({\ttfamily game}) para mantener la separación de responsabilidades.
\item {\ttfamily calloc} en lugar de {\ttfamily malloc} para arrays de punteros y contadores, garantizando que todos los campos queden a cero desde la creación.
\item {\ttfamily libscreen} solo admite caracteres ASCII de un byte:\+ ningún carácter multibyte (Unicode) debe usarse en {\ttfamily gdesc} ni en los buffers de pantalla.
\end{DoxyItemize}
